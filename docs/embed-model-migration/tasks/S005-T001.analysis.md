## Critical Finding: AC7 "Ingest Set" Does Not Yet Exist

**F1 — BLOCKER (planning gap)**

The task description says: "Queries synthesized by Claude from the S002 ingest fixture set (per D07); `expected_doc_ids` must reference docs actually ingested."

However:
- S002 tests use **fully mocked** ingest (`_make_doc()` returns `MagicMock` with random `uuid.uuid4()` — no stable IDs, nothing persisted to DB).
- `conftest.py` SAVEPOINT fixture IDs are test-isolation-only — rolled back after each test.
- There is no "S002 test ingest set" as a stable named corpus with fixed doc IDs anywhere in the repo.

**Resolution (apply before /implement T001):**

The fixture JSON cannot reference existing DB doc IDs because no stable eval corpus exists yet. The correct approach per spec AC7 is:

> "Every `expected_doc_ids` entry maps to a doc that **actually exists in the test ingest set**"

This means T001 must **define** the eval corpus doc IDs (synthetic, fixed UUIDs) and the harness (T003) must ingest them before running eval. The fixture JSON and the ingest corpus are **co-defined** — T001 creates both:

1. Fixed doc IDs embedded in the fixture JSON (`expected_doc_ids`).
2. A companion ingest manifest (either inline in fixtures JSON or a separate `multilingual_recall.ingest.json`) with `{doc_id, title, text, lang, user_group_id}` — one entry per doc referenced.
3. T003 harness ingests manifest docs before querying.

This pattern is standard for eval harnesses without a live corpus. It satisfies AC7 traceability: test assertions in T002 compare `expected_doc_ids` against the manifest's `doc_id` set.

## Fixture Structure Decision

Based on F1 above, the fixture JSON schema must carry doc content inline (or reference an ingest manifest) so the harness can seed the eval DB before measuring recall.

**Recommended extended schema** (adds `passage` field to each expected doc):

```json
{
  "version": "1",
  "ingest_docs": [
    {
      "doc_id": "e5000000-0000-0000-0000-000000000001",
      "title": "Generative AI Usage Guidelines — Section 1",
      "text": "...",
      "lang": "en",
      "user_group_id": null
    }
  ],
  "queries": [
    {
      "id": "q001",
      "query": "What are the rules for using generative AI at work?",
      "query_lang": "en",
      "expected_doc_ids": ["e5000000-0000-0000-0000-000000000001"],
      "category": "mono"
    }
  ]
}
```

This top-level split (`ingest_docs` + `queries`) keeps everything in one file, satisfies AC7 (expected_doc_ids ⊆ ingest_docs[*].doc_id), and lets T002 validate traceability with a pure JSON comparison (no DB, no Ollama).
