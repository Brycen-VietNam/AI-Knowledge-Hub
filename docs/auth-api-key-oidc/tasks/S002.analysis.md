## Gaps / Issues Found

### ⚠️ GAP — `is_active` missing on ApiKey model
`ApiKey` ORM model (S001-T003) does NOT have `is_active` column.
The T002 task spec says:
> `WHERE key_hash=:hash AND is_active=TRUE`

But `api_keys` migration DDL (`004_create_users_api_keys.sql`) also has no `is_active` column.
The `is_active` column exists on the `users` table, not `api_keys`.

**Resolution**: T002 must filter differently. Two options:
1. Join `users` table and check `users.is_active` (not `api_keys.is_active`)
2. Filter only on `key_hash` match, then check `user.is_active` via join

Looking at the task spec again: "raises `AUTH_INVALID_KEY`" for inactive keys — no distinction from not-found. This means option 2 (check `users.is_active` via join) satisfies the spec. T002 implementation should:
- `select(ApiKey).join(User).where(ApiKey.key_hash == key_hash, User.is_active == True)`

OR (simpler, consistent with spec intent):
- `select(ApiKey, User).where(ApiKey.key_hash == key_hash).join(User)` then check `user.is_active`

**Recommended**: join approach — check `User.is_active` since `ApiKey` has no `is_active`.

### ⚠️ NOTE — `AuthenticatedUser` from S004
T002 needs `AuthenticatedUser` from `backend.auth.types` which is created in S004.
Task spec says: use `TYPE_CHECKING` guard or string annotation.
**Resolution**: Define a minimal local dataclass or TypedDict in `api_key.py` for now, OR use a `dataclasses.dataclass` directly without importing from types.py.

The simplest approach (per task spec): define inline `@dataclass` in `api_key.py`:
```python
from dataclasses import dataclass

@dataclass
class AuthenticatedUser:
    user_id: uuid.UUID
    user_group_ids: list[int]
    auth_type: str
```
This will be replaced/imported from `types.py` in S004.
