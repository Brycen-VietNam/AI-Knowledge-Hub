**Design decision for T006:** ChangePasswordPage is the **force-change gate** (first login). It doesn't need a "current_password" field in the UI (no input — user never sees it). But it sends `current_password` to the API. After S001, `authStore.password` is gone. **New approach:** The force-change page must ask the user to enter their **temporary password** explicitly (add a hidden/visible current password field), OR the backend must accept a token-authenticated force-change with no current_password check. Check test: `ChangePasswordPage.test.tsx:L106` — test asserts `current_password: 'AdminSet123!'` is sent from `authStore.password`.

⚠️ **SCOPE CLARIFICATION NEEDED:** Force-change page decision:
- Option A: Add a "current/temporary password" input to ChangePasswordPage UI (user types it)
- Option B: Backend accepts force-change without verifying current_password (risk: if token stolen)
- Option C: Backend uses `refreshAccessToken()` silent re-login to confirm identity (complex)
**Recommend Option A** — matches UX where admin sets temp password and user types it on first login. This is the safest approach and consistent with standard password-change flows.

## Patterns to Follow

**Backend token creation** — use the inline pattern already in `auth.py:104`:
```python
jwt.encode({"sub": sub, "user_id": str(user_id), "exp": expire, "iat": now},
           secret, algorithm="HS256")
```
Extract this into `backend/auth/jwt.py` as `create_access_token()` + `create_refresh_token()`.

**Backend fail-fast on missing env var** — follow `auth.py:34–35`:
```python
if not _AUTH_SECRET_KEY:
    raise RuntimeError("Missing required env var: AUTH_SECRET_KEY")
```
Apply same pattern for `JWT_REFRESH_SECRET` in `jwt.py`.

**Route rate limiter** — follow `auth.py:40`:
```python
_rate_limiter = RateLimiter(resource="auth_refresh", limit=30, window=60)
```

**Error shape** — follow `users.py:30–31`:
```python
def _error(request_id, code, message): return {"error": {"code": code, "message": message, "request_id": request_id}}
```
For the route-level errors in `auth.py`, use `auth_error(request, code, message, status)` from `backend/auth/_errors.py`.

**Mock DB pattern for new SELECT cols** — follow `test_auth.py:62–65`:
```python
row.__getitem__ = lambda self, i: [user_id, password_hash, False, 1][i]  # extend to 4 cols
```

**Frontend authStore state** — `useAuthStore.setState({...})` for resetting in tests (see `authStore.test.ts:6–12`).

## Conflicts / Gaps Found

⚠️  **`backend/auth/jwt.py` DOES NOT EXIST** — T001 creates it; `create_access_token()` must be extracted from `auth.py:104` into this new module and imported back. Do not duplicate jwt.encode logic.

⚠️  **LoginForm.tsx L39–53: inline re-login using stored password** — This is the primary vulnerability. The entire `scheduleRefresh` callback must be replaced. After T005, `authStore.refreshAccessToken()` handles it internally with the refresh token.

⚠️  **ChangePasswordPage.tsx L14: reads `authStore.password`** — Two sub-issues:
1. `authStore.password` will be removed (T005)
2. The force-change page needs a way to send `current_password` — recommend adding explicit UI input (Option A above). Tests must be updated to match.

⚠️  **ChangePasswordModal.tsx: does not update authStore after success** — After T007 changes response to 200+tokens, the modal must call `authStore` update. Add `useAuthStore` import + token replacement in success handler.

⚠️  **`test_users.py` asserts 204** — T007 changes to 200. `_mock_db_with_hash` returns only 1 col (`password_hash`); must extend to 2 cols (`password_hash, sub`) and add a mock for RETURNING clause.

❌  **SECURITY: `LoginForm.tsx:47` passes `password` to `login()` in the refresh callback** — active re-use of plaintext password for re-auth. This is exactly DEFERRED-SEC-001 and must be eliminated.

### T003 — New `/v1/auth/refresh` route
- Add Pydantic model `RefreshRequest(BaseModel): refresh_token: str`
- Route: `@router.post("/v1/auth/refresh")` — no `verify_token` dep
- Call `verify_refresh_token(body.refresh_token)` — on `ValueError` → 401 `AUTH_TOKEN_INVALID`
- On success: extract `user_id` from payload; `SELECT id, sub, is_active FROM users WHERE id=:uid AND is_active=TRUE`; 401 on inactive/missing
- Issue new access + refresh tokens; return same shape as login
- OIDC detection: `verify_refresh_token` will raise `ValueError` for any non-HS256 token → already handled as `AUTH_TOKEN_INVALID`. No special OIDC detection needed beyond this — if token doesn't decode with `JWT_REFRESH_SECRET`, it fails. D-SA-07 satisfied.
- Rate limit: `RateLimiter(resource="auth_refresh", limit=30, window=60)` — module-level like login

### T005 — authStore.ts
- Remove `password` from `AuthState` interface and initial state
- Add `refreshToken: string | null` (initial: `null`)
- Change `login(token, username, password, mustChangePassword?)` → `login(accessToken, refreshToken, username, mustChangePassword?)`

  - Also wire refresh timer: after setting state, schedule `refreshAccessToken()` using exp from JWT decode OR pass `expiresIn` as 4th arg. **Simplest approach:** keep `scheduleRefresh` public (LoginForm calls it after login) — do not inline timer in `login()`. LoginForm passes `expSeconds` and uses `authStore.refreshAccessToken` as the callback.
- Add `refreshAccessToken: () => Promise<void>` async action:

  - If null → logout(); return
  - POST `/v1/auth/refresh` with `{refresh_token: refreshToken}`
  - On success: call `login(data.access_token, data.refresh_token, username, mustChangePassword)` + `scheduleRefresh(nextExp, refreshAccessToken)`

- **authStore tests:** rewrite all 3 tests that reference `password`. Replace with `refreshToken` equivalent. Add tests: `refreshAccessToken()` success path; `refreshAccessToken()` 401 → logout; `refreshToken` not in localStorage.
