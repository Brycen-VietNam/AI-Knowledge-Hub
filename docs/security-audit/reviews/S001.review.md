### ⚠️ WARNING — Should fix (post-merge OK)

#### W1 — D-SA-07 error code drift (documentation/spec inconsistency)
- **Where:** [backend/api/routes/auth.py#L160-L163](backend/api/routes/auth.py#L160-L163), [tests/api/test_auth.py#L437](tests/api/test_auth.py#L437)
- **Issue:** Task spec & WARM mem state OIDC tokens → `AUTH_UNSUPPORTED`. Implementation returns `AUTH_TOKEN_INVALID` (same path as expired/tampered). HOT.md was updated to match implementation, but `S001.tasks.md` T003 review criterion still says `AUTH_UNSUPPORTED`.
- **Impact:** Low — both are 401; client behavior identical. Pure spec/doc drift.
- **Fix options:**
  1. Reconcile WARM/task spec with implementation (preferred — implementation is simpler and avoids fragile "OIDC format" heuristic before decode)
  2. Add explicit OIDC detection (decode failure type-check) and emit distinct `AUTH_UNSUPPORTED`
- **Recommendation:** Option 1 — update `docs/security-audit/tasks/S001.tasks.md` T003 review criterion to `AUTH_TOKEN_INVALID`.

#### W2 — `scheduleRefresh` does not cancel existing timer
- **Where:** [frontend/src/store/authStore.ts#L60-L66](frontend/src/store/authStore.ts#L60-L66)
- **Issue:** `scheduleRefresh()` calls `setTimeout` and stores it in `_refreshTimer` without first clearing any existing timer. If `refreshAccessToken()` is invoked outside the scheduled fire path (e.g., manual call, or two near-simultaneous logins in dev), a stale timer can still fire after the new schedule.
- **Impact:** Low — would cause one extra unnecessary refresh; no security/correctness break (refresh idempotent).
- **Fix:** prepend `const { _refreshTimer } = get(); if (_refreshTimer) clearTimeout(_refreshTimer)` inside `scheduleRefresh`.

#### W3 — `RefreshRequest.refresh_token` has no max length
- **Where:** [backend/api/routes/auth.py#L57-L58](backend/api/routes/auth.py#L57-L58)
- **Issue:** Pydantic model accepts arbitrary-length strings. Combined with the 30/min rate limit this is well-bounded, but a 10MB JSON payload would still consume bandwidth + a JWT decode attempt.
- **Fix:** `refresh_token: str = Field(max_length=2048)` (typical JWT < 1.2KB; 2KB ample headroom).

#### W4 — No audit log for `/v1/auth/refresh`
- **Where:** [backend/api/routes/auth.py#L130-L189](backend/api/routes/auth.py#L130-L189)
- **Issue:** R006 strictly applies to document retrieval, not auth events. However, refresh-token reuse is a high-signal incident-response indicator. Best practice: log `request_id`, `user_id`, `client_ip` on refresh success/failure.
- **Fix (optional, defer to a security-observability story):** add `_logger.info("token_refresh user_id=%s ip=%s ok=%s", ...)`.

### 💡 Suggestion (non-blocking)
- T004 review criterion ("test decodes returned access_token and checks `tv` claim absent OR equals 1") is now stricter in code (always equals 1, never absent). Tests already enforce the stricter form — fine.
- Consider extracting the duplicated token-issuance + response-build logic from `auth.py:login`, `auth.py:refresh`, and `users.py:change_password` into a single helper (`build_token_response()`) before more callers appear.
