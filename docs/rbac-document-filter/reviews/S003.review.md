### WARNING

#### W01 — Default connection string with credentials in source (S005 minor) — FIXED
**File**: `tests/rag/conftest.py:14`
**Status**: RESOLVED (re-review 2026-04-03). Now reads `os.getenv("TEST_DATABASE_URL", "")` with empty default. No hardcoded credentials.

#### W02 — `large_seeded_session` lacks rollback on teardown
**File**: `tests/rag/conftest.py:121-185`
**Issue**: Unlike `db_session` which uses SAVEPOINT + rollback (lines 40-44), `large_seeded_session` yields the session after committing 10k rows but never rolls back. Data persists in the test database after the test run. This is acceptable for a dedicated test database that gets dropped/recreated, but could cause issues if the same DB is reused across test runs (ON CONFLICT DO NOTHING mitigates insert failures, but stale data accumulates).
**Recommendation**: Add cleanup logic after yield, or document that the test DB should be ephemeral (e.g., Docker container).

#### W03 — p95 index calculation is slightly off
**File**: `tests/rag/test_retriever_rbac.py:554`
**Issue**: `int(20 * 0.95) = 19` — this is index 19 of a 0-indexed 20-element list, which is the maximum value (p100), not p95. The standard nearest-rank method for p95 of 20 samples would be index 18 (the 19th value).
**Impact**: Low — the test is more conservative than stated (testing p100 < 1800ms rather than p95). This means it could fail on an outlier even when true p95 meets SLA.
**Recommendation**: Use `p95_idx = int(math.ceil(0.95 * len(latencies))) - 1` or `numpy.percentile`.

#### W04 — Fragile JSON serialization in large_seeded_session — FIXED
**File**: `tests/rag/conftest.py:170`
**Status**: RESOLVED (re-review 2026-04-03). Now uses `json.dumps([...])` with `import json` at top of file. No fragile `str().replace()` pattern.

#### W05 — Concurrent tests share single AsyncSession
**File**: `tests/rag/test_retriever_rbac.py:505-524`
**Issue**: `test_concurrent_no_cross_contamination` passes the same `seeded_session` (single `AsyncSession`) to all 10 concurrent `asyncio.gather` tasks. `asyncpg` does not support concurrent operations on a single connection — this may cause `InterfaceError` depending on the asyncpg version and whether the session uses a connection pool internally.
**Impact**: Test may fail intermittently under real PostgreSQL, though the RBAC logic itself is correct.
**Recommendation**: Use a session factory to create separate sessions per concurrent task, or run tasks sequentially in a loop (since the test goal is RBAC isolation, not connection concurrency).
