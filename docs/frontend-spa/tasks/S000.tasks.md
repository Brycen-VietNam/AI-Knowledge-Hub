### T002: POST /v1/auth/token route + bcrypt verify

**Status**: DONE

**File(s)**
- `backend/api/routes/auth.py` — create
- `tests/api/test_auth.py` — create

**Change description**
Create FastAPI router with `POST /v1/auth/token`. Accepts `OAuth2PasswordRequestForm` (username + password). Looks up user by username via parameterized SQL (S001). Verifies password with `passlib.CryptContext(schemes=["bcrypt"])`. On success: issue HS256 JWT `{"sub": username, "user_id": str(id), "exp": now + expire}` signed with `AUTH_SECRET_KEY`. Returns `{"access_token": ..., "token_type": "bearer", "expires_in": seconds}`. On failure: uniform 401 `AUTH_FAILED` (no username enumeration). Rate limit: 10 req/min per IP via existing SlowAPI/Valkey. Audit log AUTH_FAILED events (no password logged).

**Review criteria**
- [ ] Route is `POST /v1/auth/token` — R004
- [ ] No `Depends(verify_token)` on this route — R003 exception (like /v1/health) — D008
- [ ] SQL lookup uses `text(...).bindparams(...)` — S001 (no f-string)
- [ ] Both success and failure paths return uniform response time (no enumeration — use `secrets.compare_digest` or equivalent)
- [ ] `AUTH_SECRET_KEY` read from `os.getenv` — S005 (no hardcode)
- [ ] Audit log writes AUTH_FAILED only (no AUTH_SUCCESS needed for login endpoint, no password in log) — R006
- [ ] Test file covers: valid login → 200 + token, wrong password → 401, unknown user → 401, missing fields → 422
- [ ] TDD: test file written first (failing), then implementation

**Test command**
```bash

```

### T004: Dual-mode verify_token — HS256 local + RS256/ES256 OIDC

**Status**: DONE

**File(s)**
- `backend/auth/dependencies.py` — modify
- `tests/api/test_auth.py` — modify (add verify_token dual-mode tests)

**Change description**
Update `verify_token` in `backend/auth/dependencies.py` to handle HS256 local JWTs (issued by T002 route) before delegating to `verify_oidc_token` (RS256/ES256). New dispatch order: X-API-Key → HS256 local JWT → RS256/ES256 OIDC → 401. Add helper `_verify_local_jwt(token, db)` that decodes with `AUTH_SECRET_KEY`, validates `exp`, looks up user by `user_id` claim, and returns `AuthenticatedUser`. On any PyJWT exception → fall through to OIDC path (not raise). Only raise if OIDC also fails.
