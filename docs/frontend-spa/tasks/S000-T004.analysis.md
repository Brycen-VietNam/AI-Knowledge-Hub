## Helper: `_verify_local_jwt(token, db)` Design

```python
async def _verify_local_jwt(token: str, db: AsyncSession) -> AuthenticatedUser:
    # 1. jwt.decode(token, AUTH_SECRET_KEY, algorithms=["HS256"])
    #    options={"require": ["exp", "sub", "user_id"]}
    # 2. Validate exp (PyJWT raises ExpiredSignatureError automatically)
    # 3. DB lookup: SELECT id FROM users WHERE id = :user_id AND is_active = TRUE
    # 4. Return AuthenticatedUser(user_id=..., user_group_ids=[], auth_type="oidc")
    # On ANY jwt.PyJWTError → return None (fall through to OIDC, not raise)
```

**Key constraint**: return `None` on failure, NOT raise. The caller raises only after OIDC also fails.
This preserves backward compatibility: OIDC tokens still work even if `AUTH_SECRET_KEY` is set.

## Conflicts / Gaps Found

⚠️ `AUTH_SECRET_KEY` is loaded at module level in `auth.py` with `RuntimeError` guard.
In `dependencies.py`, it should be read with `os.getenv(...)` returning `None` if absent — HS256
path should be **silently skipped** (not crash) when only OIDC is configured.
**No RuntimeError guard in dependencies.py** — just `if not _LOCAL_SECRET: skip HS256`.

⚠️ `AuthenticatedUser.auth_type` Literal = `"api_key" | "oidc"` — use `"oidc"` for local users.
Confirmed same decision as T002 analysis.

⚠️ `user_group_ids` for local JWT users: JWT payload has no group claims.
**Resolution**: return `[]` (empty list). This is consistent with D06 (empty groups permissive).
Local users get no group-based access — acceptable for auth endpoint itself.

✅ DB lookup by `user_id` (UUID from JWT claim) — more reliable than `sub` (username) for lookup.
UUID is immutable; username could theoretically change.

✅ S001 compliance: parameterized SQL for user lookup.
✅ S002 compliance: PyJWT validates `exp` automatically with `ExpiredSignatureError`.
