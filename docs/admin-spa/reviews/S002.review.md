## Issues Found

### ⚠️ WARNING — Should fix (non-blocking)

**1. Error message strings hardcoded in UploadModal (not i18n)**
- File: [UploadModal.tsx:59-62](../../frontend/admin-spa/src/components/UploadModal.tsx#L59)
- Current: `setError('File too large')` / `setError('Upload failed. Please try again.')`
- Missing i18n keys: `upload_error_too_large`, `upload_error_generic`
- Impact: Error messages will not translate for ja/vi/ko users
- Fix: Add i18n keys to 4 locale files, use `t('documents.upload_error_too_large')` etc.
- Priority: SHOULD — UX issue, not a blocker for admin-only tool

**2. Loading state shows hardcoded "Uploading..." (not i18n)**
- File: [UploadModal.tsx:140](../../frontend/admin-spa/src/components/UploadModal.tsx#L140)
- Current: `loading ? 'Uploading...' : t('documents.upload_submit')`
- Fix: Add `upload_loading` i18n key
- Priority: SHOULD — same rationale as above

**3. Silent fetch failures in DocumentsPage give no user feedback**
- File: [DocumentsPage.tsx:51-53](../../frontend/admin-spa/src/pages/DocumentsPage.tsx#L51)
- Current: `.catch(() => { /* silent */ })` on `listDocuments`
- Concern: If the list fails to load, user sees an empty table with no indication of error
- Fix: Set a `fetchError` state and render an error banner
- Priority: SHOULD — acceptable for V1 admin tool; log to console at minimum

**4. `fileInputRef` declared but never used for programmatic reset**
- File: [UploadModal.tsx:19](../../frontend/admin-spa/src/components/UploadModal.tsx#L19)
- `const fileInputRef = useRef<HTMLInputElement>(null)` — ref attached to file input but never called
- The modal unmounts on close (early-return on `!open`), so state resets automatically — ref is harmless but unused
- Fix: Remove `fileInputRef` and its `import { useRef }` reference to keep the component clean
- Priority: MINOR — no behavioral impact
